# NexusFlow 端到端验证指南

> **给 AI 读的指令**：按顺序执行每张卡片。每张卡片是一个独立验证单元，包含前置条件、操作步骤、预期结果、失败处理。全部通过 = 系统端到端跑通。

---

# 启动前端（新终端）
cd nx_dashboard && npm run dev

## CARD-01 · 工作区创建

**目标**：建立后续所有操作的工作上下文

**步骤**：
1. 打开前端 → 侧边栏点击「工作区」
2. 点击「新建工作区」
3. 填写：名称 `e2e-test-project`，路径填一个本地存在的目录（如 `/tmp/e2e-test`）
4. 点击确认

**预期**：工作区出现在列表，自动选中，文件浏览器显示目录内容

**失败处理**：检查 `nx_api` 日志，确认 `/api/v1/workspaces` POST 返回 200

---

## CARD-02 · 需求输入与任务分解

**目标**：验证「描述需求 → AI 自动分解」链路

**步骤**：
1. 侧边栏点击「项目」或「需求分解」（v4.5 模块）
2. 输入需求：`实现一个 TODO List，支持增删改查，用 Python + SQLite`
3. 点击「分解」或「生成任务」

**预期**：生成 3-6 个子任务（设计、实现、测试等），每个任务有标题和描述

**失败处理**：检查 `/api/v1/projects` 或相关 AI 分解接口是否返回数据

---

## CARD-03 · Pipeline 执行

**目标**：验证任务进入执行队列并实际运行

**步骤**：
1. 侧边栏点击「执行」
2. 选择 CARD-02 生成的任务，或手动创建一个执行：
   - Workflow：选择已有 workflow 或创建简单的单步 workflow
   - 点击「运行」
3. 观察执行状态变化

**预期**：状态从 `pending` → `running` → `completed`，执行日志实时滚动

**失败处理**：
- 卡在 `pending`：检查 dispatcher 是否启动
- 卡在 `running`：检查 CLI 工具（claude/gemini）是否可用

---

## CARD-04 · 实时观测

**目标**：验证可观测性看板（v1.4）

**步骤**：
1. 在 CARD-03 执行过程中，打开「执行」页面
2. 观察：进度条、当前步骤、Token 消耗、耗时

**预期**：数据实时更新，不需要手动刷新

**失败处理**：检查 WebSocket 连接是否建立（浏览器 DevTools → Network → WS）

---

## CARD-05 · 断点续跑

**目标**：验证中断后从断点继续（v1.2）

**步骤**：
1. 启动一个多步骤执行（至少 3 步）
2. 在第 2 步执行中，重启后端：`Ctrl+C` 然后重新 `cargo run --bin nx_api`
3. 回到前端，找到该执行记录
4. 点击「继续」或「恢复」

**预期**：从第 2 步继续，不从第 1 步重新开始

**失败处理**：检查 `checkpoints` 表是否有数据：
```bash
sqlite3 nx_api/nexus.db "SELECT * FROM checkpoints LIMIT 5;"
```
若为空，说明 Checkpoint 写入未实现（已知问题 #5）

## CARD-06 · 质量门

**目标**：验证自动测试 + 失败拦截（v1.3）

**步骤**：
1. 创建一个带质量门的 workflow（在画布或 YAML 中配置 `quality_gate: true`）
2. 故意让某步骤输出包含错误关键词（如在 prompt 中要求输出 `ERROR: test failed`）
3. 运行该 workflow

**预期**：质量门检测到失败，执行状态变为 `failed`，不继续后续步骤

**失败处理**：检查 `/api/v1/executions/{id}` 返回的 `quality_gate_result` 字段

---

## CARD-07 · 失败自愈

**目标**：验证自动重试 + 修复（v4.2）

**步骤**：
1. 配置一个会失败的步骤（网络超时或错误命令）
2. 开启自愈选项（`auto_retry: true` 或 UI 开关）
3. 运行并等待

**预期**：失败后自动重试 1-3 次，日志显示重试记录

**失败处理**：检查 `execution_service.rs` 中重试逻辑是否触发

---

## CARD-08 · Git 集成

**目标**：验证每次执行自动 commit（v3.1）

**步骤**：
1. 确保 CARD-01 的工作区目录是 git 仓库（`git init /tmp/e2e-test`）
2. 执行一个会生成文件的任务
3. 执行完成后，在工作区的「Git」面板查看 diff

**预期**：出现新的 commit，diff 显示 AI 生成的文件变更

**失败处理**：检查 git watcher 服务是否启动，查看 `git_watcher.rs` 日志

---

## CARD-09 · 技能系统

**目标**：验证内置技能可调用（v2.4）

**步骤**：
1. 侧边栏点击「技能」
2. 找到一个预置技能（如代码审查、文档生成）
3. 点击「执行」，填写参数
4. 等待结果

**预期**：技能执行成功，输出结果显示在页面

**失败处理**：检查 `/api/v1/skills/{id}/execute` 接口响应

---

## CARD-10 · RAG 知识库

**目标**：验证知识库检索增强（v3.4）

**步骤**：
1. 侧边栏点击「知识库」
2. 上传一个文档（txt 或 md 文件）
3. 等待索引完成
4. 在对话或执行中引用该知识库

**预期**：文档被分块索引，执行时 AI 能引用其中内容

**失败处理**：检查 `knowledge/` 服务日志，确认 chunker 正常运行

---

## CARD-11 · Token/Cost 监控

**目标**：验证费用可见（v3.3）

**步骤**：
1. 完成 CARD-03 的执行后
2. 侧边栏点击「费用」或「Cost」页面

**预期**：显示本次执行的 Token 消耗、估算费用、按模型分类

**失败处理**：检查执行记录中 `token_usage` 字段是否有值

---

## CARD-12 · 多模型路由

**目标**：验证不同任务路由到不同模型（v4.1）

**步骤**：
1. 侧边栏点击「AI 设置」
2. 配置路由规则：代码任务 → Claude，文档任务 → 其他模型
3. 运行一个代码生成任务，观察实际使用的模型

**预期**：执行日志显示使用了路由规则指定的模型

**失败处理**：检查 `model_router.rs` 路由逻辑

---

## CARD-13 · 可视化画布

**目标**：验证低代码编排（v4.3）

**步骤**：
1. 侧边栏点击「画布」
2. 拖拽 2-3 个节点，连接成一个简单 workflow
3. 点击「导出 YAML」
4. 用该 YAML 创建一个执行

**预期**：YAML 格式正确，能被 Pipeline 解析并执行

**失败处理**：检查 YAML 导出格式是否符合 `core/workflow/src/parser.rs` 的解析规则

---

## CARD-14 · 触发器

**目标**：验证自动触发执行（v3.2）

**步骤**：
1. 侧边栏点击「触发器」
2. 创建一个定时触发器（如每分钟触发一次，用于测试）
3. 等待触发，观察执行列表是否自动新增

**预期**：到达触发时间后，自动创建并开始一次执行

**失败处理**：检查 `scheduler.rs` 定时任务是否注册
